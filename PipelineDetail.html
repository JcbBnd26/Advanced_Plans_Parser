Advanced Plans Parser — Complete Pipeline Reference

A comprehensive deep-dive into the 9-stage architectural/engineering plan parsing pipeline with OCR reconciliation, ML classification, adaptive learning, and LLM integration.

🔄 Pipeline Flow

INGEST→ TOCR→ VOCRPP→ VOCR Candidates→ VOCR→ RECONCILE→ GROUPING→ ANALYSIS→ CHECKS→ EXPORT

The pipeline processes construction drawings through text extraction, visual OCR, spatial clustering, semantic region detection, and automated quality checks. An ML feedback loop continuously improves accuracy based on human corrections.

Pre-Pipeline Setup
STEP 0 PipelineConfig — The Control Center

Before anything runs, a PipelineConfig (aliased as GroupingConfig) object holds ~150+ configuration parameters organized into typed sub-configs. Every stage reads the knobs it needs. The config system includes automatic migration for forward compatibility.

Configuration categories ▸
  • TOCRConfig — text layer extraction: x/y tolerance, font filtering, margin clipping, quality thresholds
  • VOCRConfig — visual OCR: resolution, confidence thresholds, tiling, engine selection
  • VOCRPPConfig — preprocessing: grayscale, CLAHE contrast, denoising, adaptive binarization, sharpening
  • ReconcileConfig — symbol injection: allowed symbols (/, %, °, ±), numeric context rules, overlap guards
  • GroupingConfig — clustering: row spacing, column detection, block height limits, notes detection
  • AnalysisConfig — region detection: legends, abbreviations, revisions, title blocks, semantic regions
  • MLConfig — machine learning: model paths, confidence thresholds, vision backbone, drift detection
  • LLMConfig — language models: provider (ollama/openai/anthropic), budget limits, token tracking
  • CandidatesConfig — VOCR candidate detection: gap multipliers, method toggles, ML classifier path
Sub-configs architecture ▸
  • Sub-configs are defined in config/subconfigs.py as dataclasses
  • migrate_config() handles version upgrades automatically
  • Configs are JSON-serializable with from_json() / to_json()
  • Pydantic schemas in validation/schemas.py enforce structure

Stage 1 — INGEST
STAGE 1 INGEST — Open PDF & Build PageContext

The PDF is opened exactly once via build_page_context(). This single-open strategy extracts all needed data (words, chars, lines, curves, rects, images) into a PageContext object that every downstream stage consumes. No subsequent stage re-opens the PDF.

  • Open the PDF using pdfplumber
  • Render the target page at the configured DPI (default 200) — this image feeds into VOCRPP → VOCR
  • Store the rendered image in memory — it also becomes the background for debug overlays later
  • The PDF file path is passed directly to TOCR — TOCR opens the PDF independently to read its hidden text layer; it does not use the rendered image
  • Record page dimensions (width × height in points, where 72 points = 1 inch)
  • This stage always runs — it cannot be skipped
PDF file + page number ➜ Rendered image (→ VOCRPP / VOCR path) ➜ PDF file path (→ TOCR text path) pdfplumber · Pillow

STAGE 2 TOCR — Read Text from PDF

Every PDF has a hidden text layer that describes the words printed on the page (put there by the CAD software that created the drawing). TOCR receives the raw PDF file path from INGEST, opens the PDF independently, and reads that text layer directly — no image involved. It cleans up the raw data, removes junk, and packages each word into a tidy box with its position, font name, and font size.

Processing steps ▸
  • Receive the raw PDF file path from INGEST (not the rendered image)
  • Ask pdfplumber for every word on the page, along with its position and font info
  • Convert each word into a standard "GlyphBox" — a rectangle with text and metadata attached
  • Clip boxes so none extend outside the page boundaries
  • Filter out words too close to the page edge (title blocks, borders)
  • Filter out words with unusually large or small font sizes
  • Optionally drop rotated / sideways text
  • Strip invisible control characters; discard words that become empty afterward
  • Normalize text — fix unusual Unicode forms, collapse extra spaces
  • Remove duplicate boxes — if two boxes have the same text and overlap heavily, keep only one
  • Calculate quality metrics: words per sq inch, garbled-text percentage, font distribution
  • Flag warnings if the page looks blank, has too many garbled characters, or falls below minimum density
What comes out?
  • A list of GlyphBox tokens — each with position, text, font name, font size, and origin = "text"
  • Page width and height in PDF points
  • A diagnostics report: token counts at every filter step, font distributions, density, quality flags
  • If extraction fails, an error message is returned and the pipeline gets an empty list so it can continue
Two modes
  • Full — all filtering, normalization, dedup, and diagnostics. Used by the batch pipeline.
  • Minimal — only clips coordinates and skips zero-area boxes. Used by viewers/overlays where speed matters.
Raw PDF file path (from INGEST) GlyphBox tokens (origin = text) pdfplumber 📄 src/plancheck/tocr/extract.py
STAGE 3a VOCRPP — Prepare Image for OCR

Construction drawings are often faint or noisy. VOCRPP enhances the rendered page image (grayscale, contrast, denoise, sharpen) before passing it to VOCR below. Every step is optional and toggled by the settings.

Processing steps ▸
  • Render a separate copy of the page at the OCR resolution
  • Grayscale — strip color noise
  • Auto-contrast — stretch brightness so faint text stands out
  • CLAHE — locally adaptive contrast (brightens dark corners)
  • Median denoise — smooth speckle while keeping edges sharp
  • Adaptive binarize — pure black-and-white per local region
  • Sharpen — crisp up letter edges
  • Save as page_XXXX_ocr_input.png in the run folder
  • Can be skipped — if OCR reconciliation is disabled, this stage is gated off entirely; VOCR then uses the raw unprocessed image instead
OpenCV · Pillow 📄 src/plancheck/vocrpp/preprocess.py
⬇ preprocessed image passes to VOCR
STAGE 3b VOCR — Visual OCR (PaddleOCR)

The PDF text layer misses symbols like /, %, °, and ± because CAD software renders them as vector graphics instead of real text characters. VOCR runs a deep-learning OCR engine (PaddleOCR) on the enhanced page image from VOCRPP to find those missing symbols by "reading" the picture the way a human would.

Processing steps ▸
  • Use the preprocessed image from VOCRPP if available, otherwise fall back to the raw rendered image
  • Convert to RGB (PaddleOCR needs 3-channel images)
  • If the image is very large, split it into overlapping tiles so PaddleOCR doesn't exceed its memory limit
  • Run PaddleOCR on each tile (or the whole image if it fits) — this is the slowest step in the pipeline
  • A heartbeat message prints every 15 seconds so you know it's still working
  • Convert each OCR detection from pixel coordinates back to PDF-point coordinates
  • Apply confidence threshold — discard readings below minimum confidence (default 0.5)
  • Filter out very short text and whitespace-only detections
  • If tiling was used, deduplicate tokens from overlapping tile regions using IoU
  • Record stats: total tokens, confidence min / max / mean, image source, effective DPI
Can be skipped?
  • Yes — gated by the settings. If visual OCR is disabled, this stage is skipped entirely
  • When skipped, the RECONCILE stage also skips (no OCR tokens to merge)
Preprocessed image (from VOCRPP) GlyphBox tokens (origin = ocr) PaddleOCR · NumPy 📄 src/plancheck/vocr/extract.py
Stage 3.5 — VOCR Candidate Detection
STAGE 3.5 VOCR Candidates — Intelligent Patch Selection

Instead of running expensive full-page OCR, 17+ specialized detection methods analyze the page to identify "candidate patches" — small regions where missing symbols are likely to exist. This targeted approach reduces OCR time by up to 90% while maintaining accuracy.

🎯 Three-Level Adaptive Confidence System

Level 1 — Method Stats

Per-method hit/miss rates from historical runs. Methods that perform well get higher confidence; poor performers are down-weighted. Persisted to candidate_method_stats.json.

Level 2 — ML Classifier

A trained GradientBoostingClassifier filters candidates using geometric + text features. Runs when vocr_cand_ml_enabled=True. Model stored in data/.

Level 3 — Producer Stats

Per-PDF-producer method effectiveness. CAD software from different vendors has different symbol-rendering patterns. Persisted to producer_method_stats.json.

Detection Methods (organized by tier) ▸
TIER 1: Exact Location
Char Encoding Failures
Detect Unicode replacement chars (U+FFFD) and unmappable glyphs at character level
TIER 1
Placeholder Tokens
Find tokens like "??", "□□", "__" that indicate OCR needed
TIER 2: Spatial Gaps
Intraline Gaps
Detect unusually large gaps within text lines that suggest missing content
TIER 2
Dense Cluster Holes
Grid-based density analysis to find empty pockets in dense text areas
TIER 2
Baseline Style Gaps
Gaps where baseline style changes (italic→regular) suggest symbols
TIER 3: Template Context
Template Adjacency
Patterns like "12_4" where underscore likely represents "/" or "%"
TIER 3
Regex Digit Patterns
Match "12 4", "3 8" patterns that suggest fractions (12/4, 3/8)
TIER 3
Keyword Cooccurrence
Words like "SLOPE", "GRADE" near numbers suggest % or ° symbols
TIER 3
Semantic No Units
Detect dimension-like text without unit markers
TIER 4: Advanced
Vector Circles
Detect ° degree symbols rendered as small circles in vector graphics
TIER 4
Font Subset Correlation
Fonts missing certain glyphs correlate with symbol-as-graphics rendering
TIER 4
Impossible Sequences
"12-4" that should be "12/4" based on construction notation
TIER 4
Dimension Geometry
Dimension lines with arrows but missing text
TIER 4
Near Duplicate Lines
Repeated patterns like notes where one has symbol, another doesn't
TIER 4
Token Width Anomaly
Token width inconsistent with character count (likely hidden symbol)
TIER 4
Vocab Triggers
Domain vocabulary that typically appears with symbols
TIER 4
Cross-Ref Phrases
References like "SEE DWG 1/A" patterns where "/" is commonly lost
Output structure (VocrCandidate) ▸
  • x0, y0, x1, y1 — bounding box in PDF points (with padding margin)
  • confidence — adaptive confidence (0.0–1.0) from all three levels
  • trigger_methods — list of detection methods that flagged this region
  • anchor_tokens — the TOCR tokens that triggered the detection
  • outcome — after VOCR: "hit", "miss", or "error"
  • found_text — text discovered by OCR (if any)
  • found_symbol — whether a target symbol was found

STAGE 3.5b Targeted VOCR — Patch-by-Patch OCR

When candidates exist, VOCR runs in targeted mode: each candidate's bounding box is cropped from the full-page image and OCR is run on just that small patch. Much faster than full-page OCR, and each candidate's outcome is tracked for adaptive learning.

  • Crop each candidate's bbox (with padding) from the page image
  • Run PaddleOCR on the small crop (typically 50–200px wide)
  • Update candidate with outcome: "hit" if symbol found, "miss" otherwise
  • Store found text for reconciliation
  • If no candidates exist, fall back to full-page OCR
VocrCandidate patches OCR tokens per patch PaddleOCR 📄 src/plancheck/vocr/targeted.py
Between Stages — Quick Cleanup
CLEANUP NMS Prune & Skew Correction

Before the TOCR tokens move on to RECONCILE, they get a quick cleanup pass.

  • NMS prune — "Non-Maximum Suppression." If two boxes overlap a lot (above a threshold), the smaller / lower-confidence one is removed. Prevents the same word from being counted twice.
  • Skew correction (optional) — if the page was scanned crooked, estimate the rotation angle and rotate all the boxes to straighten them out. Only runs if enabled in settings.
Stage 4 — RECONCILE
STAGE 4 RECONCILE — Multi-Phase Symbol Injection

The dual-source reconciliation engine compares TOCR tokens (PDF text layer) with VOCR tokens (visual OCR) and injects only the missing special characters. A sophisticated multi-phase approach prevents false positives while maximizing recall for symbols like / % ° ±.

Phase 0: Numeric Context Gate ▸
  • Only consider symbols appearing near numbers (e.g., "12/4", "45%")
  • Rejects random slashes in headings or standalone percent signs
  • Uses regex: has_numeric_symbol_context()
Phase 1: Spatial Alignment ▸
  • Build spatial index of all PDF tokens for fast neighbor lookup
  • For each OCR token, find best-matching PDF token by IoU or center proximity
  • Classify each match into Case A, B, or C
Phase 2: Symbol Injection Cases ▸
  • Case C (Composite) — OCR spans multiple PDF tokens. Use digit anchoring and slot-based placement to determine symbol position between tokens.
  • Case A (Extra Symbols) — OCR overlaps PDF token but contains extra symbol. Place symbol adjacent to existing word.
  • Case B (Unmatched) — OCR found symbol in whitespace. Check for digit neighbor to the left; inject if confirmed.
Phase 3: Overlap Guard ▸
  • Before injecting any symbol, verify it doesn't overlap existing tokens
  • After all injections, run NMS prune again to catch any new overlaps
  • Accepted symbols become GlyphBox tokens with origin="ocr"
Candidate Stats Feedback ▸
  • After reconciliation, compute hit/miss rates per detection method
  • Update candidate_method_stats.json (Level 1 adaptive)
  • Update producer_method_stats.json (Level 3 per-vendor)
  • Optionally save outcomes to CorrectionStore for classifier training
TOCR + VOCR tokens Merged tokens + injection log NumPy · Regex 📄 src/plancheck/reconcile/
Stage 5 — GROUPING
STAGE 5 GROUPING — Geometry-First Clustering

Transform a flat pile of word boxes into structured layout: rows → lines → spans → blocks → notes columns. Pure geometry-based (no ML) — uses positions, spacing, and font analysis.

Clustering pipeline ▸
  • group_rows() — sort by Y, cluster into horizontal rows
  • split_wide_lines() — break lines spanning multiple columns at largest gaps
  • build_lines() — combine nearby rows into Line objects with reading order
  • split_line_spans() — break lines into Span groups at large horizontal gaps
  • group_blocks() — stack nearby lines vertically into BlockCluster objects
  • mark_tables() — flag blocks with columnar alignment as tables
  • mark_headers() — identify header rows (bold, larger font, top of block)
  • mark_notes() — detect numbered-note patterns (1. ..., 2. ..., etc.)
  • group_notes_columns() — combine consecutive note blocks under shared headers
  • link_continued_columns() — connect "NOTES" to "NOTES (CONT'D)"
Output data structures ▸
  • GlyphBox — single token with bbox, text, font, origin
  • Span — sequence of tokens on same baseline
  • Line — one or more spans forming a logical line
  • RowBand — horizontal slice containing tokens at similar Y
  • BlockCluster — group of lines forming a paragraph/table/section
  • NotesColumn — notes blocks with header and continuation links
Merged token list Blocks · Lines · NotesColumns NumPy 📄 src/plancheck/grouping/clustering.py

STAGE 6c Region Detectors (Sequential Pipeline)

Specialized detectors run in sequence. Each detector adds its area to an exclusion mask so subsequent detectors don't double-count regions.

  • ① AbbreviationRegion — two-column lists (CODE = MEANING). Returns parsed entries.
  • ② MiscTitleRegion — page-level titles ("OKLAHOMA DEPARTMENT OF TRANSPORTATION")
  • ③ RevisionRegion — revision tables with number, description, date per row
  • ④ Filter Graphics — remove shapes inside already-detected regions
  • ⑤ LegendRegion — symbol + description pairs; scans remaining graphics
  • ⑥ StandardDetailRegion — sheet-number / description reference tables
  • ⑦ TitleBlockInfo — parse title block fields (project name, sheet number, etc.)
  • ⑧ SemanticRegion — header-anchored content areas with confidence scores

A persistent SQLite database (data/corrections.db) stores all pipeline detections, human corrections, and training examples. Forms the foundation for continuous learning.

Database schema ▸

ML LOOP ElementClassifier — Block Type Classification

A GradientBoostingClassifier trained on corrected examples to predict block element types (header, notes, table, legend, abbreviations, etc.). Uses 40+ engineered features.

Feature categories ▸
  • Geometric — x/y position, width/height, aspect ratio, zone position
  • Text — font size, is_bold, is_all_caps, token count, avg word length
  • Content — keyword patterns (NOTES, LEGEND, etc.), digit presence
  • Relational — below header?, header distance, sibling count
  • OCR — mean/min token confidence
Training & calibration ▸
  • Balanced class weighting via sample_weight
  • Ensemble mode: GBM + HistGradientBoosting (optional LightGBM/XGBoost)
  • Post-training isotonic calibration for well-calibrated probabilities
  • Model stored in data/element_classifier.pkl
📊 GradientBoostingClassifier 📄 src/plancheck/corrections/classifier.py

ML LOOP ML Feedback Application

On each pipeline run, apply prior corrections and ML predictions. Three-pass approach: prior corrections → ML relabeling → confidence scoring.

  • Pass 1 — Match prior corrections to new detections by bbox IoU
  • Pass 2 — ML relabeling: if model disagrees with rule-based label and confidence ≥ threshold, update
  • Pass 3 — Write model confidence to all detections
  • Pass 4 — Drift detection (optional): flag feature vectors outside training distribution
📄 src/plancheck/ml_feedback.py
ML LOOP Active Learning

Rank pages by model uncertainty to prioritize annotation. Pages with high mean uncertainty (1 - max confidence) should be reviewed first for maximum training value.
