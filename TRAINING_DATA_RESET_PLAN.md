# Training Data Reset — Implementation Plan

## Overview

**Goal:** Add a "Reset Training Data" button that clears all ML training signal from the database while preserving structural configuration, so the user can start fresh with the new online learning workflow.

**Why reset instead of letting the model self-correct:** With 179 training examples, even a small number of poisoned rows (bad features from incorrect grouping) have outsized influence. The online learning workflow will rebuild that 179-row dataset in one or two plan set reviews — roughly 30 minutes. Clean data from the start means the model converges faster and doesn't fight conflicting examples.

---

## What Gets Cleared vs. What Stays

This is the most important part of the plan. Getting this wrong either loses valuable configuration or leaves poisoned data behind.

### Cleared (ML training signal)

These tables and files contain data derived from human corrections, model training, or ML predictions. All of it gets rebuilt naturally through the online learning workflow.

| Target | Type | Why Clear It |
|--------|------|--------------|
| `corrections` table | DB | The source of all training data. Bad corrections = bad training examples. |
| `training_examples` table | DB | Generated from corrections by `build_training_set()`. Rebuilds automatically. |
| `training_runs` table | DB | Historical training metrics. Meaningless once the underlying data is gone. |
| `feature_cache` table | DB | Cached feature vectors tied to old detections. Stale after reset. |
| `candidate_outcomes` table | DB | VOCR candidate hit/miss data. Tied to old pipeline runs. |
| `dismissed_detections` table | DB | User dismissals. Tied to old detections that won't exist after reset. |
| `box_groups` table | DB | User-defined groupings. Tied to old detection IDs. |
| `box_group_members` table | DB | Group membership. Orphaned without box_groups. |
| `detections` table | DB | Pipeline detections with features. Old detections may have bad grouping-derived features. User will re-run the pipeline to generate fresh ones. |
| `training_data.jsonl` | File | Exported training data. Regenerated by `export_training_jsonl()`. |
| `training_data_stage2.jsonl` | File | Stage-2 training data. Regenerated during retrain. |
| `element_classifier.pkl` | File | The trained model. No training data = no model. |
| `drift_stats.json` | File | Drift reference. Meaningless without a corresponding model. |
| `metadata` session_active flag | DB | Clear to prevent a stale session-active flag from blocking auto_retrain. |

### Preserved (structural configuration)

These tables and files represent configuration and document registration that took effort to set up and have no ML signal in them.

| Target | Type | Why Keep It |
|--------|------|-------------|
| `documents` table | DB | Document registrations (doc_id, filename, path, page count). Re-registering PDFs is unnecessary friction. |
| `processing_runs` table | DB | Pipeline run history. Useful for provenance even after training reset. No ML signal. |
| `page_images` table | DB | Rendered page PNGs for offline viewing. Expensive to regenerate, no ML signal. |
| `label_registry.json` | File | Custom label definitions with colors, descriptions, patterns. This represents intentional configuration work. |
| `candidate_method_stats.json` | File | VOCR method performance stats. Useful reference, not training signal. |
| `producer_method_stats.json` | File | VOCR producer stats. Same reasoning. |
| DB schema | DB | All table structures. The reset is a data operation, not a schema operation. |

### Why We Clear Detections Too

This deserves an explanation because it's the least obvious decision. Detections come from the pipeline, not from user corrections. Why not keep them?

Because **features are baked into detections at pipeline time.** If the grouping was wrong during that pipeline run, the detection's `features_json` contains bad zone data, bad aspect ratios, bad text density. Those features become the input vector for ML training. Keeping old detections with bad features defeats the purpose of the reset.

The user re-runs the pipeline after reset anyway — that's step 1 of the online learning workflow. Fresh pipeline run = fresh detections = clean features.

## Implementation

### Change 1: New Method — `CorrectionStore.reset_training_data()`

**Location:** `src/plancheck/corrections/store.py`

**What it does:** Clears all ML training signal in a single transaction. Creates an automatic snapshot first (so the user can undo if needed).

**Key behaviors:**
- Takes a snapshot tagged `"pre-reset"` before doing anything destructive
- Clears tables in the correct order to respect foreign key constraints
- Deletes associated files (`*.jsonl`, `*.pkl`, `drift_stats.json`)
- Returns a summary of what was cleared (row counts) for the UI to display
- Runs `VACUUM` afterward to reclaim disk space (the DB can shrink significantly)

**Foreign key ordering matters.** The tables have foreign key relationships:

```
box_group_members → box_groups → detections → documents
corrections → detections → documents
training_examples → corrections → detections
feature_cache → detections
dismissed_detections → detections
candidate_outcomes (no FK constraints on other cleared tables)
```

Delete order (children first):
1. `box_group_members`
2. `box_groups`
3. `feature_cache`
4. `dismissed_detections`
5. `training_examples`
6. `corrections`
7. `candidate_outcomes`
8. `detections`
9. `training_runs`
10. `metadata` (clear session_active only)

**Why not just delete and recreate the DB?** Because we're preserving `documents`, `processing_runs`, and `page_images`. Those tables have real data the user shouldn't have to rebuild.

### Change 2: New Method — `CorrectionStore.get_training_data_summary()`

**Location:** `src/plancheck/corrections/store.py`

**What it does:** Returns a dict with counts from every table that would be affected by a reset. Used by the confirmation dialog so the user knows exactly what they're clearing.

**Returns:**
```python
{
    "corrections": 47,
    "training_examples": 179,
    "training_runs": 3,
    "detections": 412,
    "feature_cache": 380,
    "candidate_outcomes": 0,
    "dismissed_detections": 5,
    "box_groups": 12,
    "box_group_members": 28,
    "model_file_exists": True,
    "drift_stats_exists": True,
    "jsonl_exists": True,
}
```

This lets the confirmation dialog say: "This will clear 47 corrections, 179 training examples, 412 detections, and delete the trained model. A snapshot will be created first."

### Change 3: GUI Button — `_on_reset_training_data()`

**Location:** `scripts/gui/mixins/model_training.py`

**What it does:** Handles the UI flow for training data reset.

**Flow:**
1. Call `get_training_data_summary()` to get counts
2. Show a confirmation dialog with the specific counts
3. If user confirms, call `reset_training_data()`
4. Delete the model file, JSONL files, and drift stats
5. Reload classifiers (which will now find no model)
6. Refresh stats display
7. Show status: "Training data reset. Snapshot saved as pre-reset. Run the pipeline to begin fresh training."

**Confirmation dialog text should be explicit and specific.** Not "Are you sure?" but rather:

```
Reset Training Data

This will permanently clear:
  • 47 corrections
  • 179 training examples  
  • 412 detections
  • 3 training runs
  • The trained model (element_classifier.pkl)
  • Drift statistics

A snapshot will be saved automatically before reset.
You will need to re-run the pipeline before training.

Documents and label registry are preserved.

Continue?
```

### Change 4: Button Placement — `tab_annotation.py`

**Location:** `scripts/gui/tab_annotation.py`, in the Train & Evaluate section

**Placement:** In the secondary button row (alongside Metrics, History, Importance), not in the primary row with Train/Bootstrap/Training Session. This is a destructive action — it shouldn't be next to the buttons you click frequently.

**Tooltip:** "Clear all corrections, training examples, detections, and the trained model. Creates a safety snapshot first. Use this when starting fresh with the online learning workflow."

## File Cleanup

The reset method handles the DB tables, but several files on disk also need cleanup:

| File | Action |
|------|--------|
| `data/element_classifier.pkl` | Delete |
| `data/element_classifier.pkl.bak` | Delete (if exists — leftover from rollback) |
| `data/training_data.jsonl` | Delete |
| `data/training_data_stage2.jsonl` | Delete |
| `data/drift_stats.json` | Delete |
| `data/subtype_classifier.pkl` | Delete (if exists — Stage-2 model) |
| `data/snapshots/*` | **Preserve** — these are the user's safety net |

The file deletion should happen in the GUI method (`_on_reset_training_data`), not in the store method. The store handles its own data (SQLite). The GUI knows about the broader file layout. This keeps responsibilities clean.

## Safety Measures

### Automatic Pre-Reset Snapshot

Before clearing anything, the method calls `self.snapshot(tag="pre-reset")`. This creates a timestamped copy of the full database in `data/snapshots/`. The user can always restore from this snapshot using the existing Restore Snapshot button.

This is the undo mechanism. No need to build anything new — the snapshot/restore infrastructure already works.

### Confirmation With Specific Counts

The dialog shows exactly what will be deleted, not a vague warning. The user sees their actual data counts and makes an informed decision.

### No Silent Data Loss

The status bar confirms what happened and reminds the user of the next step: "Training data reset. Snapshot saved. Run the pipeline to begin fresh training."

---

## What Happens After Reset

The user's workflow after hitting Reset:

1. **Status:** No model, no corrections, no detections. Documents and labels are intact.
2. **Next step:** Go to Pipeline tab, re-run the batch pipeline on a plan set.
3. **Result:** Fresh detections with clean features populate the DB.
4. **Next step:** Go to ML Trainer, start a Training Session, begin page-by-page review.
5. **By page 5-10:** Model is trained from scratch on clean data with the online learning loop.
6. **Outcome:** Cleaner baseline than the old data, faster convergence, no competing bad examples.

## Implementation Order

### Step 1: `get_training_data_summary()`
Pure read-only query. Safe to build and test without any risk.

### Step 2: `reset_training_data()`
The core clearing logic. Test standalone: call it, verify tables are empty, verify preserved tables still have data, verify snapshot was created.

### Step 3: `_on_reset_training_data()` in model_training mixin
Wire the confirmation dialog and file cleanup. Test with the GUI.

### Step 4: Button in tab_annotation.py
Place the button and tooltip. Test end-to-end: click Reset → confirm → verify clean state → re-run pipeline → start training session.

---

## What We're NOT Changing

- **`reset_training_data()` does NOT delete documents, processing_runs, or page_images.** Document registration is preserved.
- **`label_registry.json` is NOT touched.** Label definitions are configuration, not training signal.
- **Snapshots directory is NOT cleared.** That's the user's safety net.
- **The reset does NOT trigger a retrain.** There's nothing to train on. The model file is deleted. The classifier will report "No model trained" until the user generates new corrections.
- **The schema is NOT modified.** No migrations. Tables are emptied, not dropped and recreated.
